Задача: влить ветку origin/refactor/core-cohort-sessions в integration-branch. 
Ожидаются серьёзные конфликты в handlers/cohorts.py и core/db/cohorts_repository.py, 
потому что обе ветки независимо резали один и тот же исходный монолитный файл 
в разных местах, начиная от разных точек истории.

ГЛАВНЫЙ ПРИНЦИП разрешения конфликтов: ничего не терять из обеих сторон. 
Конкретно:

1. core/db/cohorts_repository.py — в integration-branch это полный файл 
   (CRUD когорты, участники, посещаемость, verify_cohort_owner, 32 
   owner-check'а по всему домену). В core-cohort-sessions это тонкий файл 
   только с verify_cohort_owner. Результат слияния должен быть полной 
   версией из integration-branch — но ОБЯЗАТЕЛЬНО проверь, что сигнатура 
   verify_cohort_owner(cohort_id, uid) в обеих версиях совпадает (принимает 
   те же аргументы, возвращает то же), потому что core/db/cohort_sessions_repository.py 
   (из core-cohort-sessions) вызывает эту функцию и рассчитывает на 
   конкретную сигнатуру.

2. handlers/cohorts.py — сравни обе версии функция за функцией. Из 
   integration-branch уже вынесены (и не должны там остаться дублями): 
   создание/чтение/архивация когорты, участники, посещаемость. Из 
   core-cohort-sessions уже вынесены (и не должны остаться дублями): 
   создание/чтение/обновление/удаление сессий, регулярные правила. 
   Результат слияния должен содержать handlers/cohorts.py, где ОБЕ группы 
   функций вынесены в core/, а сам файл — только обработчики Telegram, 
   вызывающие core.services.cohorts и core.services.cohort_sessions.

3. core/db/cohort_sessions_repository.py и core/services/cohort_sessions.py — 
   эти файлы существуют только в core-cohort-sessions, конфликта не должно 
   быть, просто добавляются. Но проверь, что импорт verify_cohort_owner 
   в них указывает на актуальный (полный) core/db/cohorts_repository.py 
   после слияния, а не на несуществующую старую версию.

4. КРИТИЧНО: не потеряй ни один из 32 owner-check'ов из integration-branch 
   и ни один owner-check из когортных сессий (finalize_schedule, 
   finalize_recurring_schedule, csdt_/cstp_/cslk_/csdl_/csdy_/cspz_/csrl_/csry_/catt_mark) 
   при разрешении конфликтов в handlers/cohorts.py — это код безопасности, 
   его пропуск при слиянии эквивалентен откату уже закрытой уязвимости.

После разрешения конфликтов:
1. Запусти бота, убедись что стартует без ошибок импорта.
2. Явно перепроверь (построчно, как делали раньше) сохранность всех 
   owner-check'ов — не полагайся на то, что git merge их сохранил 
   автоматически только потому что не показал конфликт в этом месте.
3. Дай отчёт: список файлов с конфликтами, как каждый разрешён, и 
   явное подтверждение "все owner-check'и на месте" с указанием, как 
   именно это проверено (не просто заявление).

Не переходи к дальнейшей работе (перенос других доменов) — сначала полностью 
завершить и проверить эту интеграцию, включая полный прогон чек-листов 
безопасности из предыдущих отчётов по когортам.
