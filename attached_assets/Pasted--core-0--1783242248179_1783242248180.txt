2. Найди в database.py все функции, связанные конкретно с таблицей clients (CRUD: создание, чтение, обновление, архивация, удаление клиента, работа с тегами, если они в той же таблице).
3. Найди во всех handlers/*.py файлах места, где эти функции вызываются — составь список (файл, номер строки/функции, что вызывается).

5. Выведи весь список текстом. ОСТАНОВИСЬ, дождись моего "продолжай".

ЭТАП 1 — Создание структуры (после подтверждения)
1. Создай папки: core/, core/db/, core/services/, core/utils/.
2. core/utils/time.py — перенеси сюда функцию конвертации UTC → часовой пояс пользователя (ту, что уже есть/недавно чинили), плюс now_utc()/now_str() из database.py. database.py должен теперь импортировать их отсюда (from core.utils.time import ...), а не иметь свою копию — чтобы не было двух версий одной и той же функции в разных местах.
3. core/db/base.py — базовая функция подключения к SQLite (aiosqlite.connect к prokhor.db), которую будут использовать репозитории ниже. Не переноси сюда всё содержимое database.py целиком — только сам механизм подключения.

ЭТАП 2 — Перенос домена "Клиенты"
1. core/db/clients_repository.py — перенеси сюда ВСЕ функции работы с таблицей clients из database.py (создание, чтение одного/списка, обновление, архивация, удаление, теги) — без изменения их внутренней логики, просто перенос + использование core/db/base.py для подключения.
2. core/services/clients.py — если в текущих функциях в database.py или в handlers/clients.py есть бизнес-правила (например, "нельзя создать клиента, если превышен лимит тарифа", "при удалении клиента также удалить его сессии/заметки") — вынеси эти правила сюда как отдельные функции, которые внутри себя вызывают core/db/clients_repository.py. Если правил нет и там просто прямой CRUD без доп.логики — этот файл может быть тонким, просто переэкспортировать repository, не выдумывай бизнес-правила, которых не было.
3. В database.py удали перенесённые функции (не оставляй дублирующиеся копии) — но оставь остальные функции (сессии, тарифы, когорты и т.д.) как есть, их пока не трогаем.
4. Во всех handlers/*.py, где вызывались старые функции из database.py для работы с клиентами — поправь импорты на core/services/clients.py (или core/db/clients_repository.py, если это простой CRUD без бизнес-правил).

ЭТАП 3 — Отчёт и чек-лист для ручной проверки
Выведи:
1. Список всех изменённых файлов.
2. Diff-сводку: сколько функций перенесено, сколько строк убрано из database.py и handlers/*.py.
3. Чек-лист для меня, что проверить руками в реальном боте, минимум:
   - Создать нового клиента → появился в списке
   - Открыть карточку клиента → все поля на месте (имя, теги, дата создания)
   - Отредактировать клиента (если есть такая функция) → изменения сохранились
   - Архивировать клиента → пропал из активного списка, не удалился физически (проверить прямым запросом к БД, что запись осталась)
   - Разархивировать (если есть) → вернулся в список
   - Найти клиента через существующий поиск (если есть) → находится
   - Пригласительная ссылка на клиента (если привязана к этой сущности) → всё ещё работает
   - Удаление клиента как часть удаления всей практики психолога (полное удаление аккаунта) → клиент реально пропадает из БД

Не переходи к переносу других доменов (сессии, заметки, тарифы и т.д.) в рамках этой задачи — это следующие отдельные шаги после того, как я подтвержу, что этот перенос прошёл чисто.
