Задача: добавить скрытую административную панель в Telegram-бота (aiogram 3, aiosqlite, SQLite), доступную только владельцу бота по его telegram_id. Панель — inline-кнопки, многоуровневая, только для чтения статистики + несколько управляемых действий с обязательным подтверждением. Работай поэтапно, после каждого этапа коротко отчитайся об изменённых файлах.

ЗАФИКСИРОВАННЫЕ РЕШЕНИЯ:
- Команда входа: /admin. НЕ регистрировать её в setMyCommands (не должна быть видна в автодополнении команд у обычных пользователей).
- Доступ проверяется по существующему ADMIN_ID / _parse_admin_id() из main.py — переиспользовать эту же логику, не писать новую проверку админа с нуля.
- ВАЖНО: проверка admin_id должна стоять не только на входе в /admin, но и в каждом callback-хендлере админ-панели отдельно (на случай если callback_data утечёт/будет переслан кем-то другим) — если проверка не проходит, тихо отвечать на callback без действия, не показывать данные и не выполнять действие.
- Все SQL-запросы для статистики — строго read-only (SELECT), кроме явно описанных управляемых действий ниже.
- Разрушительные/массовые действия (рассылка, выдача/отзыв тарифа) — только через экран подтверждения "Да, выполнить" / "Отмена", не одним тапом.
- Все действия админа (выдача тарифа, рассылка, отзыв тарифа) логировать в новую таблицу admin_actions_log.

ЭТАП 0 — Аудит перед изменениями
1. Найди в main.py текущую реализацию _parse_admin_id() и admin_giveplan — переиспользовать их логику/паттерны, не дублировать.
2. Проверь структуру таблиц users/psychologists, clients, cohorts, sessions, cohort_cases (супервизия) — какие поля есть для подсчёта статистики (даты регистрации, тариф, expires_at и т.д.), чтобы писать точные SQL-запросы, а не гадать по названиям.
3. Проверь, есть ли уже механизм хранения FSM-состояния для многошаговых админских сценариев (SQLiteFSMStorage, если этот фикс уже применён — используй его).
4. Выведи результат аудита текстом до написания кода.

ЭТАП 1 — Точка входа и главный экран панели
1. Новый файл handlers/admin_panel.py (или расширение существующего admin-related файла, если он уже есть — проверь на этапе 0).
2. Хендлер команды /admin: проверка admin_id, при неудаче — просто игнорировать команду молча (не отвечать вообще, не палить что команда существует) или ответить нейтральным "неизвестная команда" — на твой выбор, но не показывать намёк на существование админки.
3. При успехе — inline-клавиатура главного экрана с разделами: "📊 Статистика", "💳 Подписки", "🔍 Найти пользователя", "🗂 Когорты", "📢 Рассылка", "⚙️ Система". Каждая кнопка — callback_data вида admin:section_name.

ЭТАП 2 — Раздел "Статистика" (read-only)
Экран со сводкой одним сообщением (не отдельные запросы на каждую метрику пользователю, а один запрос набора метрик и один текстовый вывод):
- Психологов всего, новых за последние 7 и 30 дней
- Клиентов всего
- Когорт всего, участников когорт всего
- Сессий: за сегодня, за последние 7 дней, всего
- Активных записей в fsm_storage прямо сейчас (если таблица существует — как прокси для "сколько человек в процессе диалога")
- Размер файла БД в МБ (os.path.getsize на путь к файлу SQLite)
Кнопка "Назад" в главное меню панели.

ЭТАП 3 — Раздел "Подписки"
1. Экран с распределением пользователей по тарифам (Start/Pro/Pro+ если Pro+ уже есть в схеме, иначе Start/Pro) — просто COUNT(*) GROUP BY.
2. Подраздел "Истекают в ближайшие 7 дней" — список telegram_id/username и даты истечения, с пагинацией по 10 записей на сообщение (кнопки "Далее"/"Назад" если записей больше), учитывая лимит Telegram на длину сообщения.
3. Подраздел "Выдать/отозвать тариф вручную" — переиспользовать существующую логику admin_giveplan, но встроить как FSM-сценарий внутри админ-панели: запросить telegram_id, запросить тариф и срок, показать экран подтверждения с итоговыми параметрами, только после "Да" — применить и записать в admin_actions_log.

ЭТАП 4 — Раздел "Найти пользователя"
1. FSM-сценарий: попросить ввести telegram_id или username.
2. Найти пользователя в БД, показать карточку: тариф, дата регистрации, кол-во клиентов у него, дата последней активности (если такое поле трекается где-то в проекте — проверь на этапе 0; если нет — пропусти этот пункт карточки, не выдумывай несуществующее поле).
3. Кнопки на карточке: "Выдать Pro" (ведёт в тот же флоу выдачи тарифа с уже подставленным telegram_id), "Написать" (просто подсказка/заготовка, не обязательно реализовывать полноценную переписку).
4. Если пользователь не найден — понятное сообщение об этом, не ошибка/краш.

ЭТАП 5 — Раздел "Когорты"
Список когорт с количеством участников (с пагинацией по аналогии с этапом 3), и отдельно — общее число кейсов супервизии, добавленных за последнюю неделю (если таблица кейсов уже существует — используй её, не изобретай новую).

ЭТАП 6 — Раздел "Рассылка"
1. Выбор аудитории: Все / только Start / только Pro (и Pro+ если применимо).
2. Ввод текста рассылки (FSM, следующий шаг после выбора аудитории).
3. Экран подтверждения: показать количество получателей и превью первых ~200 символов текста, кнопки "Да, отправить" / "Отмена".
4. При подтверждении: рассылка идёт с паузой между отправками (например, asyncio.sleep(0.05) между сообщениями, чтобы не упереться в лимиты Telegram API на частоту), каждая отправка в try/except (недоступный пользователь не должен прерывать рассылку остальным — по аналогии с фиксом в reminder_loop). В конце — сводка админу: "отправлено X из Y, ошибок Z".
5. Записать факт рассылки в admin_actions_log (кто, когда, аудитория, количество получателей, первые символы текста).

ЭТАП 7 — Раздел "Система"
- Аптайм процесса (время с момента старта, хранить старт в переменной при запуске main.py).
- Дата/время последнего успешного бэкапа БД, если механизм бэкапов уже реализован (проверь на этапе 0; если бэкапов ещё нет — просто не показывай этот пункт, не выдумывай).
- Текущий journal_mode БД (SELECT из PRAGMA) — просто чтобы визуально подтвердить, что WAL применился.

ЭТАП 8 — Таблица admin_actions_log
CREATE TABLE IF NOT EXISTS admin_actions_log (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  action_type TEXT NOT NULL,      -- 'give_plan', 'revoke_plan', 'broadcast'
  details TEXT,                    -- краткое описание/параметры действия
  created_at_utc TEXT NOT NULL
)
Добавить в тот же DDL-блок init_db(), что и остальные таблицы.

ЭТАП 9 — Проверка граничных случаев
- Кто-то не-админ прислал boтe /admin — ничего не происходит, никакой утечки информации о существовании панели.
- Рассылка на аудиторию из 0 пользователей (например, "только Pro+" когда никого ещё нет на этом тарифе) — экран подтверждения должен явно показать "0 получателей" и по возможности не давать подтвердить пустую рассылку.
- Поиск пользователя по несуществующему telegram_id — понятная ошибка, не краш.
- Два раздела статистики запрошены подряд быстро (двойной тап) — не должно быть гонки состояний в FSM, использовать текущий SQLiteFSMStorage корректно.

Не трогай существующую логику admin_giveplan вне рефакторинга под FSM-сценарий панели — переиспользуй её код, а не переписывай заново. Все тексты в админ-панели можно оставить на русском без интеграции в translations.py — это интерфейс только для тебя, локализация не нужна.
