Задача: добавить в Telegram-бота (aiogram 3, aiosqlite) раздел "Меню клиента" как ПОСТОЯННУЮ кнопку в главном меню — показывается всем без исключения, независимо от того, есть ли у пользователя своя практика психолога. Если у пользователя нет ни одной связи как клиент — раздел открывается с понятным пустым состоянием, а не скрывается и не ломается. FAQ в это меню не включать. Работай строго поэтапно, после каждого этапа останавливайся и жди моего подтверждения "продолжай" перед следующим.

ЗАФИКСИРОВАННЫЕ РЕШЕНИЯ:
- "Меню клиента" — обычная постоянная кнопка в главном меню психолога, под существующими 4 разделами (Individual/Cohorts/Summary/Settings), отдельным визуальным блоком снизу. НЕ городить логику "определить роль пользователя и показать разное меню" — кнопка есть всегда, у всех, без условий.
- FAQ в это меню НЕ добавлять.
- Ничего не менять в существующих 4 разделах и их содержимом.
- Не трогать логику /start и определение "кто есть кто" — это отдельная кнопка в уже существующем меню, а не альтернативный вход.

ЭТАП 0 — Аудит (обязательно первым, код не менять, дождаться моего подтверждения после вывода)
1. Проверь, как в БД определяется связь "клиент — психолог" (таблица clients, поле client_tg_id/аналог, привязка к psych_id). Уточни, может ли один telegram_id быть клиентом у нескольких разных психологов одновременно.
2. Проверь, есть ли уже способ клиенту посмотреть список СВОИХ предстоящих сессий по запросу (не через напоминание). Если нет — отметь как отсутствующее.
3. То же самое для домашних заданий (список выданных, отметка выполнения) и чек-инов (история прошлых ответов, доступная по запросу).
4. Проверь текущий флоу самозаписи (booking) — как выглядит вход через /start book_slug, и есть ли способ записаться к УЖЕ привязанному специалисту без новой ссылки.
5. Проверь, где хранится часовой пояс клиента (из booking-флоу) — переиспользовать это поле для "Настроек клиента".
6. Выведи результат аудита текстом. ОСТАНОВИСЬ и дождись моего "продолжай".

ЭТАП 1 — Клавиатура и точка входа (после подтверждения Этапа 0)
1. В main_menu_keyboard (существующая функция) добавь пятую кнопку "🙋 Меню клиента" отдельным рядом снизу, под текущими 4 разделами. Не меняй сами 4 раздела.
2. Новая функция client_menu_keyboard(lang) в keyboards.py, по аналогии со стилем existing individual_menu_keyboard:
   - 📅 Мои сессии
   - 📚 Мои домашние задания
   - ✅ Чек-ины
   - 🔗 Записаться к специалисту
   - ⚙️ Настройки
   - "Назад" — возврат в главное меню.
3. Новые ключи переводов в translations.py (RU и EN) для всех подписей — не хардкодь текст.
4. Покажи получившийся код этой части до перехода дальше.

ЭТАП 2 — Раздел "Мои сессии"
1. Если у пользователя нет ни одной связи как клиент ни у одного психолога — показать понятное сообщение: "У вас пока нет специалистов в боте. Если у вас есть ссылка-приглашение от психолога — используйте её, либо нажмите «Записаться к специалисту»" — с кнопкой, ведущей в Этап 5.
2. Если связь с одним психологом — сразу список предстоящих сессий (дата/время в часовом поясе клиента, если уже сохранён; если нет — сначала спросить, переиспользуя логику из booking-флоу).
3. Если связей с несколькими психологами — сначала выбор специалиста (по display_name/имени), потом список его сессий.
4. Переиспользуй существующие функции получения сессий клиента из database.py, где возможно, вместо новых SQL-запросов с нуля.

ЭТАП 3 — Раздел "Мои домашние задания"
1. Список выданных заданий со статусом, возможность отметить выполненным (переиспользуй существующую логику, если она уже где-то есть для клиентской стороны).
2. Пустое состояние — то же дружелюбное сообщение, что в Этапе 2, если специалистов нет вообще.
3. Группировка по специалисту, если их несколько.

ЭТАП 4 — Раздел "Чек-ины"
1. Если есть непройденный чек-ин — показать его первым с возможностью сразу ответить.
2. История прошлых ответов — список с датами (только чтение своей же истории).
3. Группировка по специалисту, если их несколько. Пустое состояние — как в Этапе 2.

ЭТАП 5 — Раздел "Записаться к специалисту"
1. Если уже есть привязанные психологи — список для выбора, к кому записаться (переиспользуя существующий флоу самозаписи, просто с новой точкой входа).
2. Возможность ввести/вставить ссылку-приглашение на нового специалиста прямо здесь (переиспользуй разбор payload "book_slug" как обработку введённого текста).
3. Специалист не найден по введённому — понятная ошибка, не крash.

ЭТАП 6 — Раздел "Настройки" клиента
1. Часовой пояс — переиспользуй существующее поле и логику из booking-флоу, не создавай новое.
2. Простой тумблер уведомлений, только если это несложно встроить без риска сломать reminder_loop — иначе пропусти и явно объясни почему.

ЭТАП 7 — Проверка граничных случаев
- Пользователь без единой связи как клиент — все разделы "Меню клиента" должны показывать дружелюбное пустое состояние, а не ошибку/пустой список без объяснения.
- Пользователь с несколькими специалистами — группировка не путает данные одного специалиста с другим.
- Психолог удалил клиента (через починенное ранее полное удаление практики или архивацию) — при следующем заходе в "Меню клиента" эта связь корректно пропадает.
- Двойной тап по любой кнопке раздела быстро подряд — не создаёт гонку состояний в FSM.

Не трогай существующие клавиатуры/хендлеры психолога (Individual/Cohorts/Summary/Settings) и не переписывай /start — только добавь кнопку в главное меню и всё, что за ней стоит. Все новые тексты — через translations.py. После каждого этапа явно указывай изменённые файлы.
