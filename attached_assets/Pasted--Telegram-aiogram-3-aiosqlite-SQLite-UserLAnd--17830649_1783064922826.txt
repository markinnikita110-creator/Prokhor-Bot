Задача: добавить в Telegram-бота (aiogram 3, aiosqlite, SQLite на мобильном хостинге через UserLAnd) систему самозаписи клиентов на сессии через публичную карточку психолога с deep link. Работай поэтапно, после каждого этапа кратко отчитайся об изменённых файлах перед переходом к следующему.

ЗАФИКСИРОВАННЫЕ РЕШЕНИЯ (не переспрашивать, не менять по своей инициативе):
- Слоты показываются клиенту в его часовом поясе (с конвертацией из часового пояса психолога).
- Горизонт записи вперёд: 14 дней от текущей даты.
- Минимальное время до брони: без ограничения (MVP) — можно бронировать даже ближайший слот.
- Отмена/перенос уже забронированной сессии клиентом самостоятельно НЕ реализуется в этой версии — только психолог отменяет вручную через уже существующий интерфейс управления сессиями. Не добавляй клиенту кнопку отмены/переноса.
- Часовые пояса храни как IANA-имена (например, "Europe/Moscow"), не как фиксированный офсет — иначе будут ошибки при переходе на летнее/зимнее время.
- Фича доступна только на Pro-тарифе (использовать существующий механизм из plan_limits.py, по аналогии с тем, как гейтятся другие Pro-фичи).
ЭТАП 0 — Аудит перед изменениями (обязательно первым, код не менять)
1. Найди текущий обработчик CommandStart в main.py/handlers/ — как парсится payload из deep link (`/start <payload>`). Проверь, не используется ли payload уже для чего-то (промокоды, реферальные ссылки, инвайты в когорту — судя по структуре, invite_link для клиента уже может использовать похожий механизм). Нужно расширить обработку без поломки существующих кейсов — например, через префикс в payload (`book_<slug>` для записи, отличая от других форматов).
2. Проверь, как в database.py сейчас хранится и парсится created_at/expires_at (now_utc/now_str), и переиспользуй тот же подход для дат сессий.
3. Проверь, есть ли уже поле часового пояса у психолога или клиента в таблицах users/clients. Если нет — учти это в схеме этапа 1.
4. Выведи мне результат аудита текстом, прежде чем писать код дальше.
ЭТАП 1 — Схема БД (миграция, не ломающая существующие таблицы)
Добавь новые таблицы (используй CREATE TABLE IF NOT EXISTS, не трогай существующие таблицы):

booking_profile (
  psych_id INTEGER PRIMARY KEY,

Добавь поле timezone (TEXT, nullable) в существующую таблицу клиентов, если его там ещё нет — для запоминания часового пояса клиента после первого выбора, чтобы не спрашивать каждый раз.
Добавь UNIQUE constraint на таблице sessions по паре (psych_id, start_datetime_utc), если такого ограничения ещё нет — это нужно для защиты от двойного бронирования одного слота (race condition), см. этап 3. Если добавление constraint на существующую таблицу с данными технически сложно в SQLite (обычно требует пересоздания таблицы) — сделай это безопасно: создать новую таблицу с constraint, скопировать данные, переименовать, с бэкапом старой таблицы под другим именем на случай отката.
ЭТАП 2 — Настройка доступности психологом (новый раздел в разделе "Настройки" главного меню)
1. Экран "Запись клиентов" в настройках, доступный только на Pro (иначе — заглушка с предложением апгрейда, как сделано для других Pro-фич).
2. Первичная настройка: display_name, bio (с проверкой длины), автогенерация slug из display_name (транслитерация + проверка уникальности, при коллизии — добавить суффикс), выбор часового пояса психолога (список популярных IANA-зон или ручной ввод города с определением зоны, на твой выбор — но обязательно IANA-формат на выходе).
3. Настройка недельного шаблона: для каждого дня недели — вкл/выкл, время начала/окончания приёма, длительность сессии в минутах. Хранить в availability_rules (при пересохранении дня — удалять старые записи для этого weekday и вставлять новые, не накапливать дубликаты).
4. Блокировка конкретных дат (availability_exceptions, type='blocked') — выбор даты и опционально диапазона времени (пусто = блокирует весь день).
5. Тумблер booking_enabled — включает/выключает приём новых записей вообще, не затрагивая уже существующие sessions.
6. Явно проверь и подтверди в отчёте: изменение шаблона или добавление блокировки НИКОГДА не должно автоматически отменять или удалять уже существующие записи в таблице sessions. Это два независимых слоя: шаблон = что предлагать новым, sessions = что уже согласовано.
ЭТАП 3 — Публичная карточка и флоу бронирования
1. Расширь обработчик /start: если payload начинается с "book_", извлечь slug, найти psych_id по booking_profile.slug. Если booking_enabled=0 или профиль не найден — показать вежливое сообщение "запись сейчас недоступна", не крашить.
2. Показать карточку: display_name, bio, кнопка "Записаться на сессию".
3. При первом заходе клиента (если у него в таблице клиентов ещё нет timezone) — спросить часовой пояс (аналогично выбору у психолога), сохранить и переиспользовать при будущих записях.
4. Вычисление доступных слотов (функция чистой логики, без побочных эффектов, покрываемая юнит-тестом если время позволяет):
   - Взять availability_rules психолога на ближайшие 14 дней от сегодня (по датам, учитывая weekday).
   - Вычесть availability_exceptions (заблокированные даты/интервалы).
   - Вычесть уже занятые интервалы из существующих sessions этого психолога.
   - Полученные слоты сгенерировать с шагом session_duration_min + buffer_min.
   - Конвертировать время слотов из часового пояса психолога в часовой пояс клиента для отображения (использовать zoneinfo, не pytz, если возможно — уже в стандартной библиотеке Python 3.9+).
   - Слоты, которые уже наступили или начинаются в прошлом относительно текущего момента (UTC-safe), не показывать.
5. Клиент выбирает дату → слот → подтверждает.
6. При подтверждении: INSERT в sessions с привязкой к client (создать client-запись автоматически, если это новый клиент для данного психолога — по telegram_id клиента). Обернуть INSERT в try/except на IntegrityError (из-за UNIQUE constraint) — если слот заняли на долю секунды раньше, показать клиенту "этот слот только что заняли, выберите другой" и вернуть к списку слотов, обновлённому заново (не крашить, не показывать нечитаемую ошибку).
7. После успешной записи — задействовать уже существующий механизм напоминаний (reminder_loop) для этой сессии, ничего отдельно не изобретать.
8. Психологу отправить уведомление о новой самозаписи (новое сообщение, не дублировать через reminder_loop) с указанием, что сессия создана клиентом через самозапись, а не вручную — это важно, чтобы психолог видел разницу в истории.
ЭТАП 4 — Проверка граничных случаев (пройтись явно, до сдачи)
- Психолог с пустым недельным шаблоном (ничего не настроил) — карточка должна показывать "сейчас нет доступных слотов", а не падать с ошибкой.
- Клиент открывает карточку психолога, у которого booking_enabled=0 (временно выключил) — корректное сообщение, кнопка записи не показывается.
- Два клиента одновременно жмут "Записаться" на один и тот же слот — второй получает вежливый отказ и обновлённый список, не дублирование записи.
- Часовой пояс психолога и клиента разный, включая случай, когда из-за разницы поясов дата у клиента "съезжает" на соседний день относительно даты психолога — отображаемая дата должна быть в поясе клиента корректно, без рассинхрона.
- Смена часового пояса психолога после того, как у него уже настроен шаблон — время в availability_rules остаётся "как ввёл", пересчёт идёт только на этапе показа клиенту, задним числом ничего не мигрировать.
Не трогай существующие таблицы sessions/clients кроме добавления описанных полей/constraint. Не меняй существующую логику ручного создания сессий психологом. Все новые тексты для пользователя добавляй через translations.py (RU/EN), не хардкодь строки в обход существующей системы локализации.
