Задача: перенести домен "Чек-ины" (индивидуальные) в core-слой, по той же схеме, что предыдущие домены. Работай строго поэтапно, останавливайся после каждого этапа и жди моего "продолжай". Веди работу в отдельной git-ветке (например, refactor/core-checkins), не трогай main.

ЖЁСТКИЕ ПРАВИЛА (не меняются):
- Только перенос существующего кода, без изменения поведения.
- Найденные по пути баги — не чинить тихо, вынести в отчёт и спросить меня.
- Не добавлять новых функций.
- После каждого этапа — список изменённых файлов и что стало короче в старых местах.
- Использовать core/utils/time.py и core/db/base.py, уже созданные ранее — не дублировать.

ЭТАП 0 — Аудит (код не менять)
1. Найди в database.py все функции работы с чек-инами: отправка формы клиенту, сохранение ответа, чтение истории ответов по клиенту, статус "ожидает ответа" / "отвечен".
2. Найди во всех handlers/*.py, где эти функции вызываются — список (файл, функция).
3. Проверь, есть ли логика эскалации/алертов по ключевым словам в ответах на чек-ин (упоминалось в разговоре ранее как идея — проверь, реализовано ли уже хоть частично, или это только было обсуждением без кода). Если есть хоть намётки — явно опиши, что нашёл.
4. Проверь связь чек-инов с reminder_loop/APScheduler — отправляются ли чек-ины по расписанию автоматически, или только вручную психологом. Если по расписанию — это точка пересечения с будущим переносом напоминаний, отметь явно.
5. Проверь клиентскую сторону (раздел "Чек-ины" в "Меню клиента", если уже начат) — учти существующий код, не оставляй необработанным.
6. Проверь прямые SQL-запросы к таблице чек-инов внутри хендлеров, минуя database.py.
7. Выведи результат текстом. ОСТАНОВИСЬ, дождись "продолжай".

ЭТАП 2 — Перенос
1. core/db/checkins_repository.py — перенеси все функции работы с таблицей чек-инов из database.py, используя core/db/base.py. Без изменения логики.
2. core/services/checkins.py — вынеси найденную логику (эскалация/алерты, если есть, правила отправки), которая внутри вызывает core/db/checkins_repository.py. Если правил нет — тонкая обёртка.
3. Удали перенесённые функции из database.py.
4. Поправь импорты во всех handlers/*.py (включая клиентскую сторону, если она уже написана) — на core/services/checkins.py.
5. ВАЖНО: если чек-ины запускаются автоматически из main.py (внутри reminder_loop или отдельным циклом) — поправь и там импорт на core/services/checkins.py, но саму логику планирования (когда именно запускать) пока не переноси — это будет отдельный шаг при переносе напоминаний/сессий, сейчас трогаем только "что происходит при отправке чек-ина", а не "когда".

ЭТАП 3 — Отчёт и чек-лист ручной проверки
Список изменённых файлов и diff-сводка, плюс чек-лист:
- Отправить чек-ин клиенту вручную → клиент получил форму
- Клиент отвечает на чек-ин → ответ сохранился, статус изменился на "отвечен"
- Психолог видит историю ответов клиента → отображается верно, в правильном порядке
- Если есть алерты по ключевым словам — тестовый ответ с триггер-фразой вызывает уведомление психологу как раньше
- Автоматическая отправка чек-инов по расписанию (если реализована) — всё ещё срабатывает в обычном режиме, не сломалась после переноса
- Полное удаление практики психолога → ответы на чек-ины клиента реально удаляются вместе с остальным

Не переходи к переносу других доменов в рамках этой задачи.
